# Repository Guide for Coding Agents

## Scope and purpose

These instructions apply to the entire repository. This project is a maintained Homebridge platform plugin that exposes SmartThings devices to HomeKit. It is published and installed as `homebridge-smerdthings` while deliberately retaining the original Homebridge platform identifier, `HomeBridgeSmartThings`, so existing configurations remain compatible.

Keep the plugin conservative. Home automation commands can affect locks, doors, heating, alarms, and other physical devices. Reliability, bounded failure behavior, backward compatibility, and protection of private household data are more important than aggressive retries or broad refactors.

## Repository map

* `src/platform.ts` owns platform startup, discovery, filtering, registration, and cached accessory reconciliation.
* `src/multiServiceAccessory.ts` coordinates one SmartThings device and its HomeKit services, status cache, polling, commands, and offline recovery.
* `src/services/` contains capability-to-HomeKit service adapters. Service reads should go through `BaseService.getStatus()` and the owning `MultiServiceAccessory`.
* `src/smartThingsClient.ts` is the only place that should define SmartThings HTTP policy: authentication headers, timeouts, pacing, concurrency, queue bounds, and rate-limit backoff.
* `src/deviceFilter.ts` implements metadata and wildcard ignore rules.
* `src/webhook/subscriptionHandler.ts` implements the optional legacy webhook client. It is separate from the SmartThings PAT API client.
* `config.schema.json` defines the Homebridge Config UI experience. Existing JSON configuration remains authoritative and must continue to work.
* `src/*.spec.ts` contains Node test-runner tests.
* `dist/` is generated by TypeScript and is not edited directly.

`BasePlatformAccessory` is retained for compatibility and should follow the same networking and failure-handling rules as `MultiServiceAccessory`, even though current discovery creates multi-service accessories.

## Compatibility invariants

Do not change these without an explicit migration plan:

* Package name: `homebridge-smerdthings`.
* Homebridge platform name: `HomeBridgeSmartThings`.
* Existing configuration keys and their meanings.
* Existing exact-name `IgnoreDevices` behavior.
* Minimum supported runtime declared in `package.json`.

New configuration fields must be optional, have safe defaults, appear in `config.schema.json`, and leave old copied configurations valid. Avoid storing derived private device metadata in configuration or source fixtures.

## SmartThings networking rules

All SmartThings API calls must use the shared client from `platform.smartThingsClient`. Do not create independent Axios clients in accessories or services.

Preserve these safety properties:

* Every request and queue wait is bounded.
* Requests are globally paced and concurrency-limited.
* Device status is cached for at least 10 seconds. This keeps routine reads below SmartThings' documented per-device request limit.
* Polling phases are staggered; do not synchronize every device at startup.
* HTTP 429 handling honors `X-RateLimit-Reset` and `Retry-After` before falling back to exponential backoff.
* Temporary connection failures, HTTP 408, 429, and 5xx responses are treated as transient.
* HTTP 401 and 403 responses must not create repeated per-device request storms.
* SmartThings discovery follows pagination and retries only transient failures.
* HomeKit read handlers return or fail within Homebridge's handler deadline while a safe background refresh may continue.
* Consecutive failure counters reset only after successful requests.

Never automatically retry a device command. A timed-out command may have succeeded remotely, and replaying it could duplicate a lock, door, switch, or thermostat action. Commands should return failure promptly, clear their in-progress state in `finally`, and invalidate cached status after confirmed success.

Use SmartThings health calls only for low-frequency recovery. Do not restore the per-device startup health burst.

When changing API behavior, verify current recommendations against official SmartThings developer documentation. Rate-limit headers returned to PAT clients report `X-RateLimit-Reset` in milliseconds; parse defensively and cap unreasonable waits.

## Polling and webhook behavior

Polling is the default for PAT-based configurations. A configured `WebhookToken` disables polling and uses the existing webhook handler. Do not silently migrate users to an OAuth/API Access App flow: that is a separate integration architecture requiring explicit user setup and credential migration.

Polling code may update several HomeKit characteristics from one cached SmartThings device status. Avoid creating one SmartThings status request per characteristic or service. If polling changes, measure both startup bursts and steady-state request volume across multi-service devices.

## Device filtering

Ignore rules are alternatives: a device is ignored when any rule matches. Within one rule, every configured field must match. Wildcards are case-insensitive, with `*` matching any number of characters and `?` matching one character.

Keep filtering metadata-driven and generic. Never commit real device IDs, installed app IDs, endpoint app IDs, access tokens, household names, bridge identifiers, or device labels obtained from a live Homebridge system. Tests must use obviously synthetic values.

`LogDeviceData` is intentionally opt-in because its output may contain private identifiers. Any new diagnostic logging should be disabled by default and documented with its privacy implications.

## Development workflow

Install dependencies with:

```sh
npm install
```

Before committing, run all of:

```sh
npm test
npm run build
npm run lint
git diff --check
```

Tests use the built-in Node test runner with TypeScript registration. Add focused regression tests for filtering, parsing, backoff classification, scheduling, or other logic that can be isolated. For networking changes, test failure and timeout paths as well as successful responses.

Use the Homebridge logger rather than `console`. Keep TypeScript compatible with the repository's compiler target and lint configuration. Prefer small, explicit state transitions and `try`/`catch`/`finally` over nested promise chains.

Do not edit generated `dist/` files. `npm run build` recreates them, and package installation runs the `prepare` script.

## Live validation

Live Homebridge access is not implied by repository access. Only connect to, modify, restart, or deploy to a Homebridge host when the user explicitly authorizes it.

When authorized, validate with a locally packed tarball before publishing:

```sh
npm pack --pack-destination /tmp
```

Keep remote configuration out of the repository. Do not print or copy tokens into logs, commands, patches, test fixtures, commit messages, or issue text. Redact device names and identifiers when reporting results.

After deployment, verify at minimum:

* The expected plugin version loads.
* The child bridge starts and remains running.
* Its configured port is listening.
* Discovery succeeds.
* Polling produces no 429 storm.
* There are no slow Homebridge read-handler warnings.
* There are no unhandled rejection or child-exit messages.

If a new build begins flooding SmartThings or destabilizing Homebridge, disable only this plugin and restore the previous known-good package/configuration.

## Public-repository privacy and security

This repository is public. Before every commit, inspect the complete diff for:

* Access and webhook tokens.
* Device, location, room, installed app, endpoint app, and bridge IDs.
* Hostnames, usernames, addresses, household names, and real device labels.
* Remote configuration files, logs, backups, package tarballs, and cached accessories.

Do not commit data copied from a live system even if it appears harmless. Use synthetic fixtures such as `device-123`, `Example Sensor`, or `endpoint-example`.

Do not add telemetry, analytics, credential forwarding, or new third-party network services. Any new external endpoint or credential flow requires explicit user approval and documentation.

## Versioning and releases

Use semantic versioning. Update both `package.json` and `package-lock.json`, add concise release notes to `README.md`, and run the full verification suite before creating a release commit or tag.

For an authorized release:

1. Set the version without creating a tag yet: `npm version <version> --no-git-tag-version`.
2. Update release notes and run tests, build, lint, and `git diff --check`.
3. Inspect the diff for private data.
4. Commit the release.
5. Create an annotated `v<version>` tag.
6. Push the commit and tag only when explicitly requested.
7. If live deployment is authorized, install that exact packed version and verify the runtime version.

Never move or recreate a published version tag silently. If code changes after packaging or tagging, create a new patch version.

## Git hygiene

Preserve unrelated user changes. Do not use destructive reset or checkout commands. Keep commits narrowly scoped and describe behavioral changes rather than implementation trivia.

Do not push, tag, publish, deploy, open a pull request, or modify a live system unless the user has authorized that action. When direct-to-`master` maintenance is authorized, still complete tests and the privacy review before pushing.
